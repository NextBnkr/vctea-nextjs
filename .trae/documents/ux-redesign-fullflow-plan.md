# VC查 全链路 UX 重构计划（输入页 + 结果页）

## 1) Summary
- 目标：将当前“可用但粗糙”的流程重构为移动端优先、强引导、对融资新手友好的完整体验，覆盖输入页 `/` 与结果页 `/result`。
- 设计方向：温暖引导；交互节奏为“每步 2-3 题”；结果页优先提升“可比较性”。
- 视觉原则：减少硬边框、强化留白与层次呼吸感，采用柔和曲线与微动效建立高级感。
- 硬约束：workflow 输入字段键名保持不变（如 `sai_dao`、`rong_zilun_ci` 等）；保留并升级“自动填写历史记录”能力。

## 2) Current State Analysis

### 2.1 输入页（`app/page.tsx`）
- 已具备分步向导雏形（4 步）、本地草稿与历史记录、基础校验。
- 主要问题：
  - 视觉层级与信息节奏仍偏“表单堆叠”，关键引导文案分散，认知负担仍高。
  - 步骤导航可点跳转但缺少“完成状态/未完成状态”反馈，路径感不足。
  - 字段解释尚未形成统一模式（定义、示例、反例、填写策略），学习成本高。
  - 历史记录块占据主流程注意力，入口优先级需要降级为“辅助能力”。

### 2.2 结果页（`app/result/page.tsx`）
- 已具备：运行进度条、节点文本、卡片化输出与 fallback JSON 展示。
- 主要问题：
  - “可比较性”不足：目前卡片仅并列展示，缺少横向对比骨架（维度一致、排序依据、结论聚合）。
  - 关键信息优先级不清晰：用户难以快速得出“先联系谁、为什么”。
  - 输出映射可扩展性有限：下一轮 schema 变化时需更清晰的适配层。

## 3) Assumptions & Decisions
- 已确认决策：
  - 覆盖范围：输入页 + 结果页全链路。
  - 视觉气质：温暖引导。
  - 向导节奏：每步 2-3 题。
  - 结果页目标：可比较性优先。
  - 字段约束：提交 schema 键名不变。
  - 端侧策略：移动端优先。
  - 历史能力：保留并升级（不打断主流程）。
  - 视觉反馈：减少边框密度，增加留白、曲线与动画过渡。
- 外部依赖：
  - 结果页最终 schema 将在下一轮提供；本次先搭建可比较框架与可配置映射层接口，待 schema 到位后仅替换映射配置。

## 4) Proposed Changes

### A. 输入页重构（`app/page.tsx`）
- 目标：将页面组织为“任务型向导”，让融资小白逐步完成，不需要理解术语也能填对。
- 改动要点：
  1. **布局重构**
     - 顶部从“品牌区”改为“任务引导区”：目标说明、预计耗时、当前进度、完成收益。
     - 主体采用“问题卡片流”，每步固定 2-3 个问题，减少滚动与视觉噪音。
     - 用背景分层、阴影层与透明度替代大面积边框；仅在交互焦点处使用细线边界。
  2. **步骤系统升级**
     - 步骤状态改为 `未开始 / 进行中 / 已完成`，并显示已完成数量与总量。
     - 限制跳步策略：允许回退；前跳时对未完成必填项给予明确定位提示。
     - 步骤切换加入平滑过渡（淡入 + 轻微位移），减少突兀跳变。
  3. **字段解释模板统一**
     - 每个关键字段统一结构：`这项是什么` + `如何填写` + `示例` + `常见误区`。
     - 融资金额与 runway 使用高可读说明卡，避免术语误解。
  4. **历史记录降噪升级**
     - 从主流程显性块改为“次级入口”（可折叠/抽屉式），默认不抢主视觉焦点。
     - 保留一键回填、删除、清空，新增“最近一次回填提示”。
  5. **移动端优先细节**
     - 扩大可点击区域、缩短单屏内容长度、固定底部操作区（上一步/下一步/提交）。
     - 文案长度与换行策略面向小屏优化。
  6. **视觉语言升级**
     - 组件形态采用圆角曲线体系（大圆角卡片 + 胶囊按钮 + 软进度条）。
     - 保持“少边框 + 大留白 + 明确焦点色”原则，避免“表格化、工具化”观感。

### B. 结果页重构（`app/result/page.tsx`）
- 目标：让用户先得到“谁更值得优先沟通”的答案，再查看细节依据。
- 改动要点：
  1. **比较优先的信息架构**
     - 新增“比较总览区”：按统一维度并排展示机构（匹配度、结论、主要风险、建议动作）。
     - 支持默认排序（按匹配度/结论等级）与“首选建议”标签。
     - 比较项采用“软分割”而非重边框：留白分组 + 层级对比 + 色块强调优先级。
  2. **分层阅读**
     - 第一层：结论摘要与行动优先级。
     - 第二层：每家机构展开详情（亮点/风险/补充材料）。
     - 第三层：原始输出（调试与兜底）。
  3. **映射适配层**
     - 将“输出字段 -> UI 维度”映射抽成集中配置（在当前文件内先落地为常量映射函数）。
     - 下一轮拿到 schema 后只改映射，不改页面骨架。
  4. **状态体验优化**
     - 运行中强化可感知反馈：当前阶段 + 最近节点 + 预计下一步说明（文案级，不引入后端依赖）。
     - 异常态给出明确返回动作（返回修改、重试评估）。
  5. **动效与过渡**
     - 结果加载采用骨架屏/渐显过渡，卡片进入使用轻微上浮动效。
     - 按钮、进度与展开收起统一 easing 与时长，确保节奏一致。

### C. 一致性与可维护性
- 在两页统一颜色层级、圆角体系、按钮语义与提示组件样式。
- 保留现有 workflow 调用方式与 sessionStorage 传递方式，不改接口行为。
- 补充统一动效规范（时长、曲线、触发时机），避免动画风格割裂。

## 5) Implementation Steps
1. 重构 `app/page.tsx` 的页面骨架与步骤组件结构（保持现有字段 state 和 schema 提交不变）。
2. 改造字段解释模块与校验反馈文案，补齐“术语解释卡 + 误区提示”。
3. 调整历史记录入口层级与交互（保留能力，降低主流程干扰）。
4. 重构 `app/result/page.tsx` 为“总览比较 + 机构详情 + 原始输出”三层结构。
5. 在 `app/result/page.tsx` 抽取输出映射配置函数，预留下一轮 schema 对接点。
6. 统一输入页与结果页的视觉 token（间距、标题层级、按钮状态、提示框样式）。
7. 落地统一微动效规范（切步、悬停、聚焦、进入、展开），确保“有动效但不过度”。

## 6) Verification
- 静态验证：
  - `GetDiagnostics` 检查 `app/page.tsx`、`app/result/page.tsx` 无新增错误。
  - `npx eslint app/page.tsx app/result/page.tsx` 通过（或仅存在仓库既有历史问题且与本次改动无关）。
- 交互验证（本地预览）：
  - 输入页：步骤切换、分步校验、术语提示、历史回填、移动端布局。
  - 提交流程：schema 键名保持不变，能正常跳转结果页。
  - 结果页：比较总览可读、卡片展开信息完整、重试与返回可用、异常提示可理解。
  - 视觉体验：边框显著减少、卡片留白充足、曲线语言一致、动效连续且不卡顿。
- 回归验证：
  - workflow 正常执行，结果可渲染；无结果时 fallback 展示正常。
