# 首页双组件降噪与后置引导 UX 方案（移动端优先）

## Summary
- 目标：解决首页「添加到桌面」与「先完善资料」展示位置不佳导致主流程分散的问题。
- 设计原则：主流程更聚焦（首屏只强化“开始查询”），次要动作延后触发（首页弱、结果页强），并通过文案降低理解成本。
- 输出深度：布局 + 文案优化，覆盖首页与结果页，兼顾现有逻辑不破坏。

## Current State Analysis
- 首页当前在同一屏内存在三个操作入口：顶部「添加到桌面」、底部「开始查询 / 先完善资料」并列，造成视觉注意力分散。相关实现位于 `app/page.tsx`：
  - 顶部安装按钮：约第 286-299 行
  - 主操作双按钮：约第 358-364 行
- 首页已具备两类弹窗能力：
  - 资料拦截弹窗（查询前）：约第 378-395 行
  - 安装说明弹窗：约第 399-417 行
  说明当前技术基础已支持“后置引导”，无需新增复杂状态体系。
- 结果页顶部已存在「添加到桌面」按钮（约第 240-247 行，`app/result/page.tsx`），天然适合承接“首页弱、结果页强”的安装引导策略。
- 当前问题本质：
  - 信息层级：次要动作与主动作处于同层，主路径不够单一。
  - 交互节奏：首页承载了过多“决策点”，用户未开始任务就被要求做额外选择。

## Assumptions & Decisions
- 已确认偏好（来自本轮对齐）：
  - 核心目标：主流程更聚焦。
  - 方案深度：布局 + 文案优化。
  - 端侧优先级：移动端优先。
  - 安装策略：首页弱化、结果页强化。
  - 完善资料策略：从并列按钮降级为输入框下方文本入口。
  - 节奏策略：接受“先体验后引导”。
- 约束：
  - 保持项目既有视觉基线（amber + slate、轻玻璃、移动优先）。
  - 不新增业务后端依赖，不改变会话/鉴权链路。

## Proposed Changes

### 1) 首页主操作区重构（聚焦单一 CTA）
**文件**：`app/page.tsx`  
**What**
- 将当前底部并列双按钮改为“单主按钮 + 文本次入口”：
  - 保留主按钮「开始查询」为唯一高对比动作。
  - 将「先完善资料」降级为输入框下方次级文本链接（非实心按钮样式）。
- 文案优化建议：
  - 次入口文案改为「资料不完整？先完善资料」或「想提高匹配准确度？先完善资料」。

**Why**
- 移除并列强动作竞争，减少犹豫成本，缩短用户进入主流程路径。
- 文本入口仍可见但不抢主焦点，兼顾“可达性”和“聚焦度”。

**How**
- 调整原 `mt-5 flex items-center gap-3` 操作区结构为单按钮容器。
- 在输入框区块下新增一行小字链接区（左对齐，弱色，hover 强化）。
- 保留原 `router.push('/intake')` 逻辑不变，仅变更展示层级。

### 2) 首页安装入口弱化（从按钮改为轻入口）
**文件**：`app/page.tsx`  
**What**
- 顶部「添加到桌面」由当前胶囊按钮降级为轻量文本入口（或 icon + text link），避免与业务主流程争夺视觉资源。
- 在已安装状态下，首页不再强调“已添加到桌面”的占位展示，仅保留轻提示（必要时可隐藏）。

**Why**
- 安装是“增强体验”动作，不应抢占首次任务入口。
- 弱化后可保留可发现性，同时避免首屏多主动作并存。

**How**
- 复用现有 `handleInstall` 与 `showInstallModal` 逻辑，不改安装能力判断分支。
- 仅替换按钮视觉层级（尺寸、背景、边框、字重）与布局位置权重。

### 3) 结果页安装引导强化（承接“先体验后安装”）
**文件**：`app/result/page.tsx`  
**What**
- 保留并强化结果页顶部「添加到桌面」入口：
  - 视觉权重提升至“次主动作”层级（高于普通返回按钮、低于核心业务动作）。
  - 增加短文案辅助（如“下次可像 App 一样一键打开”）。
- 根据状态展示差异：
  - 未安装：突出收益导向文案。
  - 已安装：显示轻量完成态。

**Why**
- 用户完成一次查询后感知价值更强，安装转化更自然。
- 与“首页弱、结果页强”策略一致，符合用户已确认偏好。

**How**
- 维持现有 `canInstall / isStandalone / installing` 状态判断，调整顶部按钮组层级与顺序。
- 在结果页标题区或进度模块下添加一条简短说明，不引入新弹窗流程。

### 4) 文案层级统一（降低认知负担）
**文件**：`app/page.tsx`, `app/result/page.tsx`  
**What**
- 按动作角色统一文案：
  - 主动作：任务导向（开始查询）
  - 次动作：收益导向（完善资料提高准确度 / 安装提升复访效率）
  - 提示态：状态导向（已安装、处理中）

**Why**
- 角色一致性能显著降低用户扫描成本，增强页面可预期性。

**How**
- 替换按钮与提示短文案，不新增业务字段或国际化结构。

## Verification Steps
- 视觉与交互验收（移动端优先，含桌面回归）：
  - 首页首屏是否仅存在一个强主动作按钮。
  - 「先完善资料」是否可见且可点击，但视觉权重明显低于主按钮。
  - 首页安装入口是否弱化但功能可达。
  - 结果页安装入口是否较首页更易感知，点击逻辑正常。
- 功能回归验收：
  - 首页主按钮跳转与拦截弹窗逻辑不回归。
  - 完善资料入口跳转 `/intake` 正常。
  - 安装逻辑在可安装/不可安装/已安装状态下均符合原有分支行为。
- 体验验收标准（本次成功判定）：
  - 首屏用户决策点减少为“先查”主路径。
  - 次要动作在不丢失可达性的前提下显著降噪。
  - 安装引导实现“首页弱、结果页强”的节奏一致性。
