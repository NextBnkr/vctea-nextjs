# VC查「机构搜索首页拆分 + 柔性拦截引导」实施计划

## 1) Summary
- 目标：将当前“长表单首页”拆成“搜索引擎式首页 + 信息采集页”，让用户先明确“查机构”任务，再进入资料采集与自动补全流程。
- 关键路径：用户在首页搜索机构后，若已填过团队信息则直接出结果；若未填则跳转 `/intake` 并带入目标机构，展示柔和的“商业机密与准确性”引导文案。
- 本期约束：候选建议通过 Next.js 假接口返回静态机构数据，不引入外部依赖或数据库。

## 2) Current State Analysis
- 当前首页 `app/page.tsx` 集成了 Hero、历史记录、4 步表单、自动补全、提交跳转与 storage 逻辑，页面职责过重。
- 当前结果页 `app/result/page.tsx` 仅依赖 `sessionStorage.vccha_run_payload`，因此只要 payload 结构保持一致即可复用。
- 当前已存在草稿/历史存储：`vccha_form_draft_v1`、`vccha_form_history_v1`；自动补全与提交流程已能跑通。
- 机构输入能力已有组件 `app/components/base/tag-input/index.tsx`，可在新信息采集页复用。
- 现有补全路由 `app/api/workflows/prefill/route.ts` 已可返回结构化字段，且主/补全双 key 已就位（`app/api/utils/common.ts` + `config/index.ts`）。

## 3) Proposed Changes

### A. 新增“搜索引擎式首页”并将原表单迁移到 `/intake`
- 文件：
  - `app/page.tsx`（重写为搜索首页）
  - `app/intake/page.tsx`（承接现有表单主逻辑）
- 改造内容：
  - `app/page.tsx` 仅保留：价值主张、机构搜索框、候选建议列表、搜索按钮。
  - 搜索提交后执行“资料是否已填”判定（核心三项：`oneLineSummary` + `fundingAmount` + `teamBackground`）。
  - 若已填：将“本次机构”覆盖到 payload 输入并直接跳转 `/result`。
  - 若未填：柔性提示“我们将用户信息视为商业机密，需基于真实上报数据给出结论”，随后跳转 `/intake`。
  - 跳转时通过 query 或 sessionStorage 传递本次机构关键词（推荐 query：`/intake?org=xxx`，同时写 sessionStorage 兜底）。

### B. 在 `/intake` 接入“搜索带入机构”与原自动补全流程
- 文件：
  - `app/intake/page.tsx`（由当前 `app/page.tsx` 迁移并轻改）
- 改造内容：
  - 保留当前多步向导、自动补全、历史与草稿逻辑。
  - 首次进入时读取 `org` 参数并注入 `targetInvestors` 初始值，避免用户重复输入。
  - 保持提交 payload 结构不变（`mi_ao_shu`、`jin_e`、`ji_gou` 等键维持现状），确保结果页零改造可用。
  - 保持当前 runway 容错映射与加载动画逻辑。

### C. 新增候选建议假接口（静态返回）
- 文件：
  - `app/api/institutions/suggestions/route.ts`（新增）
  - `service/index.ts`（新增获取建议方法）
- 改造内容：
  - 新增 GET 接口，支持 `q` 查询参数，返回静态机构列表（可按关键词过滤，最多返回 N 条）。
  - 首页输入时调用该接口渲染候选下拉/建议 chips（无需引入第三方搜索库）。
  - 若接口失败，首页降级为仅搜索框可提交，不阻断主流程。

### D. 资料判定与直达结果策略
- 文件：
  - `app/page.tsx`
  - 可选抽离：`app/lib/profile.ts`（新增判定函数）
- 判定规则（已确认）：
  - 核心三项齐全即“已填”：`oneLineSummary`、`fundingAmount`、`teamBackground` 非空。
  - 数据来源优先级：`vccha_form_draft_v1` > `vccha_form_history_v1[0]`。
- 直达结果实现：
  - 以已存资料构造 `vccha_run_payload`（沿用当前 intake 提交映射逻辑）；
  - 将本次搜索机构写入 `ji_gou`；
  - `router.push('/result')`。

### E. 文案与交互（柔性引导）
- 文件：
  - `app/page.tsx`
- 文案原则：
  - 不“拦截式警告”，而是“可信解释 + 继续动作”。
  - 说明两点：信息按商业机密处理；结论需基于用户真实上报。
- 交互形式：
  - 搜索后若未填资料，显示轻提示卡片（1-2 句）并自动进入 `/intake`；
  - 保留“返回首页”路径，避免强制感。

## 4) Assumptions & Decisions
- 决策：搜索输入采用“单机构 + 候选建议”。
- 决策：若已填团队信息，搜索后直接出结果。
- 决策：团队信息判定使用核心三项齐全。
- 决策：新采集页路由为 `/intake`。
- 决策：候选建议使用 Next.js 假接口返回静态数据。
- 决策：未填资料时直接进入 Step1，并预填目标机构。
- 新增偏好：拦截文案要柔和，强调“商业机密”与“基于真实上报”。

## 5) Verification Steps
- 路由与入口：
  - 访问 `/` 仅看到搜索式首页，不再出现完整表单。
  - 搜索任意机构后，未填资料用户进入 `/intake?org=...`，且机构已自动带入。
- 判定与分流：
  - 构造“核心三项齐全”的本地数据后，首页搜索可直接跳 `/result` 并触发工作流。
  - 核心三项缺失时不直达结果，进入 intake 继续补全。
- 建议接口：
  - `GET /api/institutions/suggestions?q=红杉` 返回静态候选（过滤后列表）。
  - 接口异常时首页仍可提交搜索并继续流程。
- 回归：
  - intake 页自动补全、runway 下拉回填、加载动画、提交到结果页流程保持正常。
  - `vccha_run_payload` 结构与结果页解析兼容，不引入额外字段破坏。
